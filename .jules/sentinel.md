## 2024-05-18 - Markdown Parser XSS Vulnerability
**Vulnerability:** The custom Markdown parser was vulnerable to Stored XSS via maliciously crafted URLs in links and images. The naive `startsWith('javascript:')` check was easily bypassed using URL encoding (`javascript%3A`), HTML entities (`javascript&#x3A;`), and embedded control characters/whitespace (`java\tscript:`).
**Learning:** Custom URL sanitizers that rely solely on string matching are notoriously difficult to get right and prone to bypasses. Browsers are highly fault-tolerant and will execute JavaScript even if the URI contains whitespace or encoded characters.
**Prevention:** Always use the native `URL` constructor to reliably extract and check the protocol. When implementing custom sanitization, explicitly decode URL components and HTML entities, and strip control characters/whitespace before validation.

## 2024-05-18 - Markdown Preview XSS Vulnerability
**Vulnerability:** A stored/reflected XSS vulnerability existed in `app/submit/page.tsx` within the Markdown preview feature. The preview took raw user input and passed it to `dangerouslySetInnerHTML` after basic string replacements, bypassing the application's secure `MarkdownRenderer` entirely.
**Learning:** Developers often implement "quick and dirty" rendering for previews, forgetting to use established secure rendering components. This creates a critical vulnerability vector because previews operate directly on untrusted user input before any backend sanitization occurs.
**Prevention:** Always reuse established, secure rendering components (like `MarkdownRenderer`) across the entire application, including temporary or preview states. Never use `dangerouslySetInnerHTML` directly with user input, even in simple preview contexts.
## 2024-05-18 - Client-Side XSS in Markdown Editor Preview
**Vulnerability:** A Stored DOM XSS vulnerability existed in `app/submit/page.tsx` where user-provided markdown content was being previewed using `dangerouslySetInnerHTML` after applying a few rudimentary regex replaces for markdown syntax. This allowed arbitrary HTML, such as `<script>` tags or `onload` handlers, to be executed in the user's browser.
**Learning:** Using `dangerouslySetInnerHTML` combined with custom, incomplete regex parsing for markdown is highly vulnerable. It does not provide adequate sanitization and allows attackers to bypass markdown rendering to inject malicious scripts. Even for "preview" functionality, the client-side execution environment is compromised.
**Prevention:** Never use custom regex replacements combined with `dangerouslySetInnerHTML` for parsing untrusted markup. Always use established, secure libraries or components (like the existing `@/components/markdown-renderer`) that handle both parsing and sanitization securely.

## 2024-05-19 - XSS in Custom Markdown Live Preview
**Vulnerability:** A Stored/Reflected XSS vulnerability existed in the live preview functionality of the document submission page (`app/submit/page.tsx`). The application used naive regex-based markdown parsing combined with `dangerouslySetInnerHTML` to render user-provided markdown content in real-time, bypassing the secure `<MarkdownRenderer>` component.
**Learning:** Developers sometimes implement ad-hoc markdown parsing for quick features (like live previews) to avoid complex integrations, inadvertently circumventing established security components and introducing XSS vulnerabilities.
**Prevention:** Always enforce the usage of secure, centralized components (like `<MarkdownRenderer>`) for rendering rich text or markdown across the entire application, and actively flag `dangerouslySetInnerHTML` usages in code reviews.

## 2024-05-19 - XSS in Search Query Headline Rendering
**Vulnerability:** A Stored/Reflected XSS vulnerability existed in the search results page (`app/search/page.tsx`). The application used `dangerouslySetInnerHTML` to render the `headline` field from PostgreSQL's `ts_headline` function. Since `ts_headline` merely wraps matching terms in `<mark>` tags and does not sanitize the surrounding context, any search results containing unescaped HTML or malicious scripts in the document text could execute in the browser.
**Learning:** Using `dangerouslySetInnerHTML` with `ts_headline` or similar database text-highlighting functions is highly vulnerable because the database functions only add formatting tags—they do not provide XSS sanitization for the rest of the content string.
**Prevention:** Never use `dangerouslySetInnerHTML` to render `ts_headline` results. Instead, parse the string manually (e.g., splitting by `<mark>` and `</mark> tags) and use React's safe rendering to output the content and the highlight tags as distinct elements, ensuring the surrounding text remains natively escaped.
